# Phase 9: Integração Frontend ↔ Backend

<objective>
Substituir a camada de dados mock pela integração real com a API REST construída nas phases anteriores. Configurar cliente HTTP, `AuthContext` com JWT, e interceptação de erros via Toasts. O frontend operará 100% com dados do backend e o banco será populado para corresponder aos mocks.
</objective>

<threat_model>
- **T-09-01:** Exposição de cookies JWT.
  - *Mitigation:* A API já usa `httpOnly` para o token. O frontend não deve tentar ler ou manipular o token JWT via JS; apenas delegar o tráfego das requests usando `credentials: "include"`.
- **T-09-02:** XSS via dados de retorno de API.
  - *Mitigation:* React já realiza escape de injeções de script nativamente. Garantir uso de `.text()` ou tipagem estrita na parse do JSON na API client.
</threat_model>

## Wave 1: Camada Base de API e Autenticação

<task>
<read_first>
- frontend/lib/api/client.ts
- frontend/types/api.ts
</read_first>
<action>
Criar/Atualizar `frontend/lib/api/client.ts` com um client de fetch universal que aceita JWT:
- Implementar as funções REST base (`get`, `post`, `put`, `del` ou `delete`).
- Toda a requisição **MANDATORIAMENTE** deve incluir:
  - `credentials: 'include'` (para trafegar os cookies `httpOnly` de Auth)
  - Prefixar as URLs com o BASE_URL (`process.env.NEXT_PUBLIC_API_URL || 'http://localhost:3001/api'`)
  - `Content-Type: application/json` quando tiver body.
- Interceptador de erro: Após `await res.json()`, se a response não for `res.ok`, extrair a mensagem (ex: `data.message`) e atirar um erro literal: `throw new Error(data.message || 'Erro na requisição')`.
</action>
<acceptance_criteria>
- `frontend/lib/api/client.ts` exporta as funções HTTP (`api.get`, `api.post`, etc).
- Todas as chamadas contém `credentials: 'include'`.
- Os erros disparam um `throw new Error(...)` extraindo a mensagem.
</acceptance_criteria>
</task>

<task>
<read_first>
- frontend/contexts/AuthContext.tsx
- frontend/mocks/session-user.ts
</read_first>
<action>
Atualizar `frontend/contexts/AuthContext.tsx`:
- Remover qualquer importe relacionado ao `mocks/session-user`.
- O método `login(email, password)` deve executar `await api.post('/auth/login', { email, password })`. Se sucesso, deve buscar o perfil via `await api.get('/auth/me')` e salvar o user no state. Se erro, atirar ou exibir o toast de erro.
- O método `register(data)` deve chamar `/auth/register` seguido de `/auth/login`.
- O método `logout()` deve chamar `/auth/logout` e limpar o context.
- O load inicial (`useEffect`) deve chamar `/auth/me` para verificar se há cookie ativo e restaurar a sessão; caso 401, definir estado de deslogado sem exibir erro via Toast.
</action>
<acceptance_criteria>
- Não existe import de `mocks` em `AuthContext.tsx`.
- O login e a checagem inicial chamam endpoints da API usando o wrapper.
</acceptance_criteria>
</task>
